# Better Working Behaviour — Software Engineer

> Monday – Friday · Daily standup 10:30

Each behaviour has a **target stat** — measurable signal you're on track. Use for self-assessment, not just a checklist.

---

## Calendar & Communication

| Behaviour | Target Stat | Current Progress |
|-----------|-------------|-----------------|
| Confirm attendance on Google Calendar (accept/decline) | 100% invite response rate | :heavy_check_mark: |
| Morning email slot (09:00–09:30) — read, triage, reply | < 24h response on items needing action | :heavy_check_mark: |
| Standup update prepared before 10:30 | 0 unprepared standups per sprint | :heavy_check_mark: |
| Defer long standup discussions to separate sync | Standup consistently < 15 min | :heavy_check_mark: |

## Sprint Planning

| Behaviour | Target Stat | Current Progress |
|-----------|-------------|-----------------|

## Mindset

| Behaviour | Growth Signal | Current Progress |
|-----------|---------------|-----------------|
| Own problems end-to-end — not just your ticket | Loops closed proactively, without being asked | :heavy_check_mark: |
| Ask "why" before "how" | Misaligned requirements caught before coding starts | :heavy_check_mark: |
| Treat code review as learning, not criticism | Constructive comments, no defensive replies | :heavy_check_mark: |
| Communicate risks and blockers early | Blockers surfaced same day, not at deadline | :heavy_check_mark: |
| Change your mind when shown better evidence | Decisions revisited when data changes | :heavy_check_mark: |
| Write code for the next person who reads it | Readability review comments decrease over time | :heavy_check_mark: |
| Ship iteratively — imperfect done beats perfect never | Features delivered in smaller, shippable increments | :heavy_check_mark: |
| Ask for help early — escalating is a skill | Time-to-escalate decreases each sprint | :heavy_check_mark: |
| Think in systems — understand how your change affects the whole | Fewer cross-team surprises per release | :hourglass: |
| Apply first principle thinking — break assumptions to fundamentals, rebuild from there | Solutions derived from fundamentals, not cargo-culted | :hourglass: |
| Apply lateral thinking — explore unconventional approaches before defaulting | Alternative approaches explored before settling | :hourglass: |
| Treat SLO as shared responsibility — actively increase and maintain reliability targets | SLO maintained or improved sprint over sprint | :hourglass: |

## Hard Rules (Non-Negotiable)

| 🔴 Rule |
|---------|
| Lunch + prayer break 12:00–13:00 — mandatory, step away from screen |
| Daily standup at 10:30 — on time, update prepared |
| No new feature development during regression phase |
| All tickets must have acceptance criteria before sprint starts |
