# Better Working Behaviour — Software Engineer

> Monday – Friday · Daily standup 10:30 · Deploy cut-off 16:30

---

## Calendar & Communication

| ✅ Good Behaviour | ❌ Bad Behaviour |
|-------------------|-----------------|
| Always confirm attendance (accept/decline) on Google Calendar invitations | Ignoring meeting invitation confirmations on Google Calendar |
| Schedule a morning slot (09:00–09:30) to open email and reply to anything needing a response | Rarely opening email — missing async updates and urgent requests |
| Prepare standup update before 10:30 — yesterday, today, blockers | Showing up to standup unprepared |
| Defer long standup discussions to a separate sync | Letting discussions drag on in standup |

---

## Sprint Planning

| ✅ Good Behaviour | ❌ Bad Behaviour |
|-------------------|-----------------|
| Read PRD/tickets before the planning meeting — come with questions and rough estimates | Starting planning without preparation |
| Ensure all tickets have acceptance criteria before sprint starts | Leaving tickets ambiguous going into dev |
| Estimate tasks before starting work | Picking up tasks without estimation |
| Push back firmly on scope creep during planning | Accepting scope creep silently — becomes a dev problem |
| If estimation debate > 10 min, create a spike item | Forcing estimation resolution on the spot |

---

## Development

| ✅ Good Behaviour | ❌ Bad Behaviour |
|-------------------|-----------------|
| Protect deep work slots (10:00–15:00) — zero notifications, zero interruptions | Allowing meetings or distractions during core coding hours |
| Escalate after 30 min stuck | Going silent when blocked |
| Open new meetings only at 15:00 or later | Scheduling meetings during deep work hours |
| PRs must have description + screenshot/video for UI changes | Raising PRs with no description or context |
| Push WIP and update ticket at end of day | Leaving teammates in the dark on task status |
| Deploy only before 16:30 | Deploying after cut-off without hotfix justification |

---

## Regression & End of Sprint

| ✅ Good Behaviour | ❌ Bad Behaviour |
|-------------------|-----------------|
| Triage and classify bugs before anything else (critical / major / minor) | Starting random bug fixes without triage |
| Fix critical bugs before demo | Demoing with known critical bugs open |
| Document root cause for every bug closed | Closing bugs without explanation |
| Demo only truly completed features to stakeholders | Demoing WIP to stakeholders |
| Document reasons for any carry-over tickets | Silently carrying over tickets with no explanation |
| No new feature development during regression | Sneaking in new features during regression phase |

---

## Hard Rules (Non-Negotiable)

| 🔴 Rule |
|---------|
| Lunch + prayer break 12:00–13:00 — mandatory, step away from screen |
| No deploys / new PRs after 16:30 except hotfix |
| Daily standup at 10:30 — on time, update prepared |
| No new feature development during regression phase |
| All tickets must have acceptance criteria before sprint starts |
